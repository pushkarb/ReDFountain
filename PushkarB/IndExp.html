  • Ecosystem - A Customer like Teradata Test Harness [2011]
    Technology: Unix Shell Scripting, Teradata Stored Procedures, User Defined Functions, Teradata Load Utilities like FastLoad, MultiLoad, TPump, TPT,

    Over a span of 3 months, I developed a test harness for testing certain aspects of Teradata Data Warehouse technology. I was the primary architect for the solution. The project involved testing Teradata Warehouses for bugs in handling the ETL processes implemented using Teradata User Defined Functions. The system simulated the workloads of some of the premier customers of Teradata and aimed to get an insight into the kind of roadblocks the customers would run into and fix them before the actual release of the software.

    I created an automated system using Unix Shell scripting, that was highly modularized making it easily scalable and also easy to understand. Through this project I also got a chance to learn about the Linux CRON scheduler. Using the Linux CRON scheduler, I ensured dataloads of varying traffic into the Teradata systems. Different kinds of workloads were scheduled through different parts of the day and of the week making it real and customer like. The application also encompassed a logging system that logged progress of each workload and provided vital information about each workload, an error detection system based on the console output files generated by individual Teradata Load Utilities, Maintenance scripts that ensured different conditions like database overflow, crash were handled. The system also ensured weekly log backup and cleanup. These were some of the features implemented into the system.

    The application loaded close to half a terabyte of data into the tables each day. During the span of one month of continuous, the application was successful in trapping three customer reported bugs that could have not been reproduced previously. The application is currently being used within Teradata with further enhancements.

Application Engineer at Oracle Corporation : June 2008 - August 2010
  • ETL Testing Tool [2009]
    Language: SQL, PL/SQL

    This was a self motivated project aimed to design a tool that could be used to test the correctness of the ETL procedures that were coded using ETL tools like Oracle Data Integrator and Informatica Power Centre. The code takes the table loaded by ETL tools and a model target table structure with correct sample test data as inputs. It would then perform a row wise comparison of all the data. The program reports success/failure if the data from the table loaded using the tool matched/mismatched with the data in the target table.

  • Benchmarks and Targets [2008-09]
    Tool: OBIEE; Language: SQL

    Benchmarks and Targets project aimed to implement an extended feature on existing OBIEE applications. Based on the requirements of the application, I designed and developed two types of Benchmarks and Targets feature. One, a global targets implementation, that doesn't depend only on the fact attribute under consideration and the time dimension, two, targets against fact attributes when atmost 3 dimensions are considered at the same time of which one of the dimensions under consideration has to be the time dimension. The project involved in designing and implementing an augmented star schema structure that would fit into the existing data model of the product.

    In case one, the structure contained a single table storing all the fact attributes against targets were to be implemented. OBIEE views are created from each row this universal table and used to store the targets.

    In case two, an OBIEE view was created for each possible combination(without repetition) of interacting dimensions with a single fact attribute. This results in a complicated structure where each benchmarks and targets view was linked to the fact from to which the fact attribute belonged to and the time dimension and the dimensions that are interacting. The way these views were positioned in the existing data model is similar to a conformed dimension.

    In both cases, all the business rules and other filter conditions were enforced in the Business Model and Mapping layer of OBIEE. Also, the level of hierarchy of the interacting dimensions upto which the targets and benchmarks would hold were also set in this layer.
